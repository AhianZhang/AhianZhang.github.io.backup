HDFS

HDFS 是 GFS 的克隆版，其有扩展性、容错性和海量存储的特点
分块：Hadoop 1.x 默认大小是 64 M，Hadoop 2.x的是 128M 每次上传文件时会进行分片多副本存储（对使用者透明，也就是对于用户来说他就是一个文件）。
副本：默认是复制 3 份

NameNode(NN)

是用来处理客户端的请求和管理 Metadata

DataNode(DN)

存储用户的文件所对应的的数据块；定期向 NN 发送心跳信息，汇报自身以及数据块的信息

HDFS 副本存放策略
  • 1st replica. 如果写请求方所在机器是其中一个datanode,则直接存放在本地,否则随机在集群中选择一个datanode.
  • 2nd replica. 第二个副本存放于不同第一个副本的所在的机架.
  • 3rd replica.第三个副本存放于第二个副本所在的机架,但是属于不同的节点.

    YARN

    Yet Another Resource Negotiator
    负责整个集群资源的管理和调度
    特点：扩展性、容错性、多框架支持
    YARN 架构
  • ResourceManange(RM)
    • 整个集群同一时间提供服务的 RM 只有一个，负责集群资源的统一管理和调度；
    • 处理客户端请求：提交一个作业、杀死一个作业
    • 监控 NM ，一旦 NM 时效，那么该 NM 上运行的任务需要告诉 AM进行如何处理
  • NodeManage(NM)
    • 整个集群中有多个，负责自己本身节点资源管理和使用，定时向 RM 汇报本节点的资源使用情况
    • 接受并执行来自 RM 的各种命令：启动 Container
    • 处理来自 AM 的命令
    • 自身单节点资源管理
  • ApplicationMaster(AM)
    • 每个应用程序对应一个AM：MR、Spark，负责应用程序的管理
    • 为应用程序向 RM 申请相应的资源（Core Memory)，分配给内部的 task
    • 需要与 NM 进行通信： 启动停止 task 它是运行在 Container 中的 ，AM 也是运行在 Container中的。
  • Container
    • 封装了 CPU Memory
    • 是一个任务运行环境的抽象
  • Client
    • 提交作业
    • 查询作业进度
    • 杀死作业

MR1.x 介绍
  • JobTracker (JT)
    • 作业的管理者
    • 将作业分解成一堆任务： Task（MapTask 和 ReduceTask）
    • 将任务分派给 TaskTracker 运行
    • 作业的监控、容错处理（task作业挂了，重启task的机制）
    • 在一定的时间间隔内 JT没有收到 TT 的心态信息，判定 TT 可能无法正常工作， TT 上运行的任务会被指派到其他的 TT 上执行
  • TaskTracker(TT)
    • 任务执行者
    • 在TT上执行 Task（MapTask 和 ReduceTask）
    • 与 JT交互 ，发送心跳 HeartBeat
  • MapTask
    • 自己开发的 Map 任务交由该 Task处理
    • 解析每条记录的数据，交给自己的 map 方法处理
    • 将 map 的输出结果写到本地磁盘
  • ReduceTask
    • 将 Map Task 输出的数据进行读取
    • 按照数据进行分组，传给自己编写的 reduce方法处理
    • 写入到 HDFS/数据库
